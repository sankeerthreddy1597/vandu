# Recipe App — Implementation Plan
> Reference document for Claude Code. Read this fully before beginning implementation.

---

## 1. Project Overview

A personal recipe-saving app for 3–5 users. Users can save recipes by:
- Sharing an Instagram Reel URL via the iOS share sheet
- Pasting any recipe URL manually
- Uploading an image of a recipe

All three flows trigger an async pipeline that scrapes/extracts recipe data, sends it to an LLM for structured output, and persists the result to the database.

| File Storage | Cloudflare R2 | For uploaded recipe images |
| Scraping | Playwright + `@sparticuz/chromium` | Deployed on Railway or Fly.io container |
| Realtime | Inngest Realtime or SSE endpoint | Status updates pushed to app |

## 4. Database Schema (Prisma)

```prisma
// packages/db/schema.prisma

generator client {
  provider = "prisma-client-js"

datasource db {
  provider = "postgresql"

  id        String   @id                // Clerk user ID
  email     String   @unique
  recipes   Recipe[]

}

model Recipe {
  id          String       @id @default(cuid())
  userId      String
  user        User         @relation(fields: [userId], references: [id])

  title       String
  description String?
  sourceUrl   String?
  sourceType  SourceType
  imageUrl    String?      // R2 URL if image was uploaded
  status      RecipeStatus @default(PROCESSING)
  error       String?      // Error message if status = FAILED

  ingredients Ingredient[]
  steps       Step[]
  tags        String[]

  createdAt   DateTime     @default(now())
  updatedAt   DateTime     @updatedAt
}

model Ingredient {
  id       String  @id @default(cuid())
  recipeId String
  recipe   Recipe  @relation(fields: [recipeId], references: [id], onDelete: Cascade)
  name     String
  amount   String?
  unit     String?
  notes    String?
  order    Int
}

model Step {
  id          String  @id @default(cuid())
  recipeId    String
  recipe      Recipe  @relation(fields: [recipeId], references: [id], onDelete: Cascade)

  instruction String
  duration    String? // e.g. "10 minutes"
  tip         String?
}

enum SourceType {
  INSTAGRAM
  URL
  IMAGE
}

enum RecipeStatus {
  PROCESSING
  SCRAPING
  EXTRACTING
  DONE
  FAILED
```

## 5. API Routes (Next.js App Router)

### `POST /api/recipe/create`
Accepts recipe creation request. Validates input, creates a `Recipe` record with `status: PROCESSING`, triggers Inngest event, returns the `recipeId` immediately.

**Request body:**
```ts
{
  type: "instagram" | "url" | "image"
  url?: string        // for instagram + url types
  imageKey?: string   // R2 object key, for image type
}
```

**Response:**
```ts
{ recipeId: string }
```

### `GET /api/recipe/:id`
Returns full recipe data including ingredients and steps.

### `GET /api/recipe/:id/stream`
SSE endpoint. Client connects here after receiving `recipeId`. Sends status update events as Inngest pipeline progresses. Closes when status is `DONE` or `FAILED`.

### `POST /api/recipe/:id/delete`
Soft or hard delete a recipe.

### `GET /api/recipes`
Returns paginated list of recipes for the authenticated user.

### `POST /api/upload/presign`
Returns a presigned R2 URL for direct image upload from the client.

### `POST /api/inngest`
Inngest webhook endpoint (auto-generated by Inngest SDK).

### `/api/auth/[...all]`
Clerk auth handler.

## 6. Inngest Pipeline

Each recipe creation triggers one Inngest function: `recipe/process`. It runs as a series of durable steps.

```ts
// apps/web/inngest/functions/processRecipe.ts

export const processRecipe = inngest.createFunction(
  { id: "process-recipe", retries: 3 },
  { event: "recipe/process" },

  async ({ event, step }) => {
    const { recipeId, type, url, imageKey, userId } = event.data

    // Step 1: Scrape or extract raw content
    const rawContent = await step.run("scrape-or-extract", async () => {
      if (type === "instagram" || type === "url") {
        return await scrapeUrl(url) // Playwright scraper service
      }
      if (type === "image") {
        return await extractFromImage(imageKey) // Claude vision
      }
    })

    await updateRecipeStatus(recipeId, "EXTRACTING")

    // Step 2: LLM structured extraction
    const structured = await step.run("llm-extract", async () => {
      return await extractRecipeWithLLM(rawContent, type)
    })

    // Step 3: Persist to DB
    await step.run("save-recipe", async () => {
      await saveStructuredRecipe(recipeId, structured)
    })

    await updateRecipeStatus(recipeId, "DONE")
  }
)
```

### Status update flow
After each step, call `updateRecipeStatus(recipeId, status)`. This function:
1. Updates the DB record
2. Broadcasts the new status to the SSE stream for that `recipeId`

## 7. LLM Extraction (Vercel AI SDK)

Use `generateObject` with a Zod schema for fully typed structured output.

```ts
// packages/types/recipe.ts
export const RecipeSchema = z.object({
  title: z.string(),
  description: z.string().optional(),
  servings: z.string().optional(),
  prepTime: z.string().optional(),
  cookTime: z.string().optional(),
  tags: z.array(z.string()),
  ingredients: z.array(z.object({
    name: z.string(),
    amount: z.string().optional(),
    unit: z.string().optional(),
    notes: z.string().optional(),
    order: z.number()
  })),
  steps: z.array(z.object({
    order: z.number(),
    instruction: z.string(),
    duration: z.string().optional(),
    tip: z.string().optional()
  }))
})

// Extraction function
async function extractRecipeWithLLM(content: string, type: SourceType) {
  const { object } = await generateObject({
    model: anthropic("claude-sonnet-4-5"),
    schema: RecipeSchema,
    prompt: `Extract a structured recipe from the following content.
    Source type: ${type}
    Content: ${content}
    
    Be precise with ingredient amounts and units.
    Split steps into clear, single-action instructions.`
  })
  return object
}
```

For **image inputs**, send directly to Claude vision:
```ts
async function extractFromImage(imageKey: string) {
  const imageUrl = getR2PublicUrl(imageKey)
  const { object } = await generateObject({
    model: anthropic("claude-sonnet-4-5"),
    schema: RecipeSchema,
    messages: [{
      role: "user",
      content: [
        { type: "image", image: new URL(imageUrl) },
        { type: "text", text: "Extract the full recipe from this image." }
      ]
    }]
  })
  return object
}
```

## 8. Playwright Scraper Service

Deploy as a **standalone long-running container** on Railway or Fly.io (not Vercel — Playwright is too large for Lambda without workarounds).

```ts
// scraper-service/index.ts (Express server)

app.post("/scrape", async (req, res) => {
  const { url } = req.body
  const browser = await chromium.launch()
  const page = await browser.newPage()

  // Set user agent to avoid basic bot detection
  await page.setExtraHTTPHeaders({ "User-Agent": "Mozilla/5.0 ..." })
  await page.goto(url, { waitUntil: "networkidle" })

  // Instagram: grab description + first author comment
  const description = await page.$eval(
    "meta[property='og:description']",
    el => el.getAttribute("content")
  )

  await browser.close()
  res.json({ content: description })
})
```

**Fallback chain:**
1. Playwright (free, self-hosted)
2. Apify Instagram Scraper (free tier: ~100 runs/month)
3. Manual entry prompt shown to user

The Inngest step should catch errors from the scraper and trigger the fallback automatically.

## 9. Cloudflare R2 Setup

```ts
// Image upload flow:
// 1. Mobile app calls POST /api/upload/presign
// 2. Server generates presigned R2 PUT URL
// 3. Mobile app uploads image directly to R2 (bypasses your server)
// 4. Mobile app sends imageKey to POST /api/recipe/create

import { S3Client, PutObjectCommand } from "@aws-sdk/client-s3"
import { getSignedUrl } from "@aws-sdk/s3-request-presigner"

const r2 = new S3Client({
  region: "auto",
  endpoint: `https://${process.env.CF_ACCOUNT_ID}.r2.cloudflarestorage.com`,
  credentials: {
    accessKeyId: process.env.R2_ACCESS_KEY_ID,
    secretAccessKey: process.env.R2_SECRET_ACCESS_KEY
  }
})

export async function getPresignedUploadUrl(key: string) {
  return getSignedUrl(r2, new PutObjectCommand({
    Bucket: process.env.R2_BUCKET_NAME,
    Key: key
  }), { expiresIn: 300 })
}
```

## 11. SSE Hook (Mobile)

```ts
// hooks/useRecipeStream.ts
export function useRecipeStream(recipeId: string) {
  const [status, setStatus] = useState<RecipeStatus>("PROCESSING")

  useEffect(() => {
    const eventSource = new EventSource(
      `${API_BASE}/api/recipe/${recipeId}/stream`
    )

    eventSource.onmessage = (e) => {
      const data = JSON.parse(e.data)
      setStatus(data.status)
      if (data.status === "DONE" || data.status === "FAILED") {
        eventSource.close()
      }
    }

    return () => eventSource.close()
  }, [recipeId])

  return status
}
```

## 13. Implementation Order

Follow this sequence to avoid blockers:

1. **Monorepo scaffold** — pnpm workspaces, Turborepo, shared packages
2. **DB + Prisma** — schema, migrations, Neon connection
3. **Next.js API skeleton** — Clerk auth, basic route structure
4. **Inngest setup** — connect to Next.js, create function scaffold
5. **LLM extraction** — `generateObject` with Zod schema, test with hardcoded input
6. **Scraper service** — Playwright on Railway, `/scrape` endpoint
7. **R2 storage** — presign flow, test upload
8. **Full pipeline** — wire Inngest steps end to end
9. **SSE endpoint** — status broadcasting
10. **Expo app** — auth, recipe list, recipe detail, SSE hook
11. **Share extension** — `expo-share-intent` integration
12. **Image upload flow** — presign + upload in app
13. **TestFlight** — EAS build config, first internal distribution

## 14. Key Decisions & Rationale

| Decision | Rationale |
|---|---|
| Inngest over raw queues | Durable steps, retries, great DX, no infra to manage |
| SSE over WebSockets | Simpler for one-way server→client status pushes |
| Playwright on Railway over Vercel | Playwright binary too large for Vercel Lambda |
| R2 over S3 | No egress fees, S3-compatible API, simpler pricing |
| Clerk over Better Auth | Better Expo SDK, zero-config for small user base |
| Presigned upload vs server proxy | Avoids routing large files through your API server |
